1. MULTI_RETURN_MUTATION: given a function with multiple returns or vararg (...) usage, modify the number of returned values — add or remove values (nil or actual), change the order, or swap for a vararg expression — keeping the expression list valid and matching the usage in the caller.

2. ARG_LIST_MUTATION: modify function calls, table constructors, multiple assignments, or local declarations by changing the number of arguments/values — inserting extra values (nil, constants) or removing some — preserving syntax and the minimum allowed number.

3. MULTI_RET_IN_SINGLE_EXPR: replace a simple expression with a call that returns multiple values, in a context where only one value is expected — leaving the expression syntactically valid (Lua adjusts extra values).

4. METATABLE_MUTATION: generate or modify metatable declarations, metamethods such as `__index`, `__newindex`, `__add`, `__concat`, `__len`, `__call`, etc. — inserting extra functions or fields, or removing some — to test metamethod fallback or unexpected behavior.

5. COMPLEX_SCOPES: mix scope styles — convert global blocks to local and vice versa, add closures, local functions, anonymous functions, with variable shadowing — causing bindings, upvalues, and possible scope conflicts.

6. ALTERNATIVE_CONTROL: transform numeric `for` loops into `for ... in ipairs/pairs`, or into `while`, or `repeat ... until`, adapting initialization, increment, and condition — maintaining similar logic but altering the control structure.

7. COMPLEX_TABLE: transform simple table constructors into tables with metatables, or with functions as values, or extra fields, or deep nesting — complex structures to test parsing and table operations.

8. BLOCK_NESTING: wrap sub-blocks of code in `do ... end`, closures, local functions, or nested `if/then` blocks, without changing the essential logic — to generate block nesting and test deep parsing.

9. DECL_ORDER_MUTATION: reorder local or function declarations (when not dependent), or interleave declarations and expressions — testing ordering, forward declarations, and possible initialization dependencies.

10. LITERAL_MUTATION: modify literals — strings (inserting escapes, special characters, unicode, newlines), numbers (floats, extremes, negatives, zero), booleans, nil — to test literal parsing and edge cases.

11. BITWISE_AND_ARITHMETIC_OPERATOR: insert bitwise operators (when applicable) or combine arithmetic and bitwise operations/concatenations between strings/numbers — to exercise type coercion and conversions.

12. SHADOW_BUILTIN: overwrite standard functions (e.g., `table.insert`, `ipairs`, `print`, etc.) with local versions (closures), and invoke both the global and local versions — testing shadowing, scope, and binding.

13. COROUTINE_MUTATION: generate functions that use coroutines — combinations of closures + coroutine + `yield` + `resume`, with parameter passing and multiple return values, to test the coroutine runtime in Lua 5.4.

14. ERROR_INJECTION: insert deliberate calls to `error()`, `assert(false)`, or invalid indexing inside blocks potentially catchable via `pcall`/`xpcall` — to test error handling, exceptions, and recovery.

15. GC_AND_METAGC: create tables with metatables containing `__gc`, eventually with finalizers, or simulate userdata usage (when applicable) — forcing garbage collection or automatic finalization, to test garbage collection and finalizers.

16. DYNAMIC_CHUNK_LOAD: create code snippets as strings and load them via `load`, `loadstring`, or `loadfile` — possibly dynamically constructed — to exercise dynamic parsing, runtime compilation, and edge cases of loading.

17. MODULES_REQUIRE_MUTATION: mix module usage with `require`, manipulate `package.path`, simulate dummy modules or mocks, use duplicate requirements or redefinitions — to test the module system, load caching, and conflicts.

18. FORMATTING_MUTATION: insert or remove comments, spaces, line breaks, irregular indentation, invisible characters (tabs, non-breaking spaces, BOM) — to test the parser's tolerance to formatting and tokens.

19. IDENTIFIER_EDGECASE: use variable / function / table field names with atypical names — long names, with underscores, unicode, repeated characters, redeclarations, shadowing, reassignments — to stress the lexer/parser.

20. MIXED_DECLARATIONS: combine different declaration styles — multiple declarations (local a, b, c = ...), simple declarations, assignments on separate lines, interleaved with blocks or expressions — to test robust parsing of diverse declarations.
